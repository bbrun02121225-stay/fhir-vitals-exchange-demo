二、學習歷程與實作心得（完整反思）

1. 從「寫得出來」到「用得起來」

一開始接觸這個競賽時，我對 FHIR 的理解主要停留在：
	•	Resource 結構
	•	API 呼叫方式
	•	如何成功 POST / GET 資料

因此我選擇用 Python 先完成一個最小可行實作，
確認資料真的能寫進 FHIR Server、也能被讀回。

這個階段，我學到的是：

FHIR 在技術上是可行的，但這只是第一步。

⸻

2. Python 的侷限，讓我重新思考「實作」的定義

在 12/17 的會議中，教授的一句回饋對我影響很大：

「這個程式目前只有你自己能用。」

我開始意識到：
	•	技術正確 ≠ 實作完整
	•	能跑 ≠ 有使用價值

如果一個系統只能在工程師電腦上執行，
那它其實還沒真正進入「應用」階段。

因此我決定捨棄「只寫 Python 就好」的舒適圈，
重新用前端的角度思考整個系統。

⸻

3. 從工程邏輯，轉換成使用者流程

將 Python 改寫為 HTML + JavaScript 的過程中，
我最大的學習不是語法，而是思維轉換。

我必須回答這些問題：
	•	使用者第一步會看到什麼？
	•	他什麼時候該輸入 Patient.id？
	•	轉出院與轉入院能不能做錯事情？
	•	系統要不要阻止錯誤流程？

這迫使我重新整理整個轉診流程，
而不只是「API 呼叫順序」。

⸻

4. FHIR 不再只是規範，而是流程的骨架

在實作前端後，我對 FHIR 的理解有明顯改變。

以前我看 FHIR，是在看：
	•	欄位
	•	JSON
	•	Schema

但現在我看到的是：
	•	誰負責建立資料
	•	誰只負責查詢
	•	資料為什麼值得被信任
	•	為什麼轉入院不應該再寫入資料

我開始理解：

FHIR 的價值，不只在資料格式，而在責任分工。

⸻

5. 從 Demo，走向「完整系統雛形」

在 12/26 與助教的討論後，
我對這個專案的定位更加清楚。

這不是一個為了展示 API 的 Demo，
而是一個：
	•	有角色區分
	•	有流程邏輯
	•	有實際臨床動機
	•	可以持續擴充的系統雛形

這也讓我在後續調整 UI、簡報與說明時，
始終以「完整系統」作為核心目標。

⸻

三、總結反思

從最初的 Python 程式，到現在可操作的前端系統，
我最大的收穫不是學會某一個技術。

而是學會：
	•	如何把技術放進真實情境
	•	如何讓系統為人服務，而不是只為工程師存在
	•	如何用 FHIR 作為流程設計的基礎，而不是限制

這次競賽對我而言，
是一個從「寫程式」走向「做系統」的轉捩點。

⸻
